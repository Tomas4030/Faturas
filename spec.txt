1. Resumo da ideia
A aplicação permite ao utilizador tirar uma fotografia a uma fatura/recibo, extrair automaticamente os
dados principais e transformar essa fatura numa experiência interativa.
Casos principais:
1. Restaurante com amigos
2. O criador tira foto da fatura.
3. A app extrai restaurante, data, itens, quantidades, preços, IVA, total.
4. O criador partilha um link com os amigos.
5. Cada amigo abre o link e seleciona os itens que consumiu.
6. A app calcula automaticamente quanto cada pessoa deve pagar.
7. Numa fase futura, a app pode facilitar pagamento via MB WAY ou gateway de pagamento.
8. Gestão pessoal de despesas
9. O utilizador digitaliza qualquer fatura.
10. A app categoriza automaticamente: restaurantes, supermercado, combustível, fornecedores,
lazer, etc.
11. A homepage mostra estatísticas por mês, fornecedor, categoria, valor gasto, evolução e gráficos.
12. Gestão para pequenos negócios/restaurantes
13. O dono digitaliza faturas de fornecedores.
14. A app mostra quanto está a gastar em cada fornecedor.
15. Ajuda a perceber onde o dinheiro está a ir.
16. Pode comparar meses, fornecedores e categorias.

2. O que construir primeiro
MVP recomendado
O erro mais comum seria tentar construir tudo ao mesmo tempo: IA, estatísticas, pagamentos,
subscrições, dashboard avançado, MB WAY, fornecedores, relatórios e gestão de equipas.
Não faças isso.
O MVP deve ser:
Foto da fatura
→ Extração dos itens
→ Ecrã para corrigir itens
→ Criar sessão de divisão
→ Partilhar link
→ Amigos escolhem itens
→ Cálculo final por pessoa
Isto já resolve uma dor real.
Só depois de isto funcionar bem é que deves avançar para:
Histórico de despesas
→ Categorias
→ Estatísticas
→ Planos pagos
→ MB WAY
→ B2B/restaurantes

4. Stack recomendada
Mobile app
Recomendação: React Native com Expo.
Motivos:
• Rápido para MVP.
• Funciona em iOS e Android.
• Bom suporte para câmara, upload de imagens, deep links e partilha.

Alternativa: Flutter.
Flutter também é excelente, mas se a equipa já souber JavaScript/TypeScript, React Native + Expo é
mais prático.

Recomendação: Node.js com NestJS ou FastAPI com Python.

Backend principal: NestJS + TypeScript

Pagamentos/subscrições: Stripe inicialmente

6. Como fazer a extração da fatura
Não confies só numa IA genérica
Este é um ponto crítico.
Faturas são difíceis porque podem ter:
• má iluminação;
• papel dobrado;
• itens cortados;
• abreviações;
• IVA misturado;
• descontos;
• gorjeta;
• vários IVA;
• quantidades;
• linhas partidas;
• totais pouco claros;
• recibos de supermercados com muitos produtos;
• faturas de restaurantes com artigos iguais repetidos;
• QR Code fiscal;
• NIF;
• ATCUD;
• dados pessoais.
A melhor abordagem é híbrida.

8. Regra crítica: validação matemática
Nunca confies cegamente no JSON da IA.
Tens de validar:
quantity × unit_price = line_total
sum(line_total) + tax - discounts + tip = grand_total
Com tolerância, por exemplo:
0.01€ a 0.05€
Porque podem existir arredondamentos.
Exemplo onde não podes errar
Fatura:
2 x Menu Sushi 18,50 = 37,00
2 x Coca-Cola 2,50 = 5,00
Total = 42,00
Se a IA devolver:
{
"description": "Menu Sushi",

13. Pagamentos MB WAY
Fase 1 — Não integrar pagamentos
No MVP, faz apenas:
"O João deve pagar 23,40€ ao Tomás."
E mostra:
Número MB WAY do Tomás: 9XX XXX XXX
Ou botão:
Copiar valor
Copiar número
Fase 2 — Deep link ou instrução de pagamento
Podes tentar abrir uma experiência que ajude o utilizador a pagar, mas não assumes que o pagamento
aconteceu sem confirmação real.
Fase 3 — Integração SIBS/MB WAY
Aqui já entras em pagamentos reais.
Pontos importantes:
• MB WAY precisa de aceitação do utilizador.
• A SIBS permite pagamentos MB WAY através de gateway.

• Precisas de conta de merchant/contrato.
• Tens de lidar com estados: pending, accepted, rejected, expired.
• Tens de lidar com webhooks/callbacks.
• Tens de guardar logs de pagamento.
Atenção regulatória
Não construas um sistema onde a tua app “segura” dinheiro de amigos e depois distribui sem
perceberes as implicações legais.
Há diferença entre:
A app calcula quanto cada pessoa deve pagar.
e
A app recebe dinheiro de várias pessoas e distribui para outra pessoa.
A segunda pode trazer obrigações regulatórias, KYC, AML, PSP/licenciamento ou dependência forte de
um parceiro de pagamentos.
Para começar, mantém-te no modelo:
A app calcula e facilita.
O pagamento acontece fora da app ou via PSP licenciado.

23. Onde o júnior pode errar
Erro 1 — Pensar que OCR é sempre certo
Problema:

Mostrar confiança por linha.

Certo:

24. Estratégia de IA
Prompt de extração
Exemplo:
Analisa esta fatura portuguesa. Extrai os dados em JSON.
Não inventes valores.
Se não tiveres certeza, usa null.
Cada item deve ter descrição, quantidade, preço unitário e total.
Confirma se a soma dos itens corresponde ao total.
Devolve warnings quando algo não bater certo.
Schema
Usa schema fixo.
Não aceites respostas livres.
Pós-processamento
Depois da IA:
1. Normalizar moeda.
2. Converter valores para cêntimos.
3. Validar totais.
4. Marcar linhas suspeitas.
5. Classificar categoria.
6. Guardar.

32. Regra de ouro
Esta aplicação só vai ter sucesso se for mais rápida do que fazer contas à mão.
Se o utilizador tiver de:
tirar foto
esperar muito
corrigir tudo
convidar amigos
explicar como funciona
ver erros
ele desiste.
O fluxo ideal tem de ser:
Foto → confirmar → partilhar → escolher → pagar
Tudo o resto é secundário.

33. Conclusão
A ideia é boa porque junta três dores reais:
1. dividir contas de forma justa;
2. evitar contas manuais em grupo;
3. guardar histórico e estatísticas de gastos.
Mas o produto deve começar pequeno.
A versão certa para começar é:
Digitalizar fatura de restaurante
+ transformar em checklist
+ partilhar link
+ calcular quanto cada pessoa paga
Depois disso, evolui para:
estatísticas
subscrições
limites por plano
pagamentos
business/restaurantes
A parte mais importante tecnicamente é:
extração confiável
validação matemática
edição manual rápida
cálculo monetário correto
Se isto estiver bem feito, o resto pode crescer por cima.
